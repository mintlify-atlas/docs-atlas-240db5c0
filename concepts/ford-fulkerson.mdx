The **Ford-Fulkerson algorithm** finds the maximum flow through a network by repeatedly discovering paths from source to sink and pushing as much flow through them as possible, until no more augmenting paths exist.

NetworkFlux & MaxFlux Generator implements the **Edmonds-Karp variant**, which uses breadth-first search (BFS) to find augmenting paths. This choice guarantees polynomial time complexity regardless of edge capacities.

## High-level algorithm

<Steps>
  <Step title="Initialize the residual graph">
    Copy the original capacity matrix into a residual graph `rGraph`. All entries start equal to their original capacities; backward edges start at 0.
  </Step>
  <Step title="Find an augmenting path">
    Run BFS from source `s` to sink `t` on the residual graph. BFS only follows edges with positive residual capacity. If no path exists, stop — maximum flow has been reached.
  </Step>
  <Step title="Compute bottleneck capacity">
    Walk the path from `t` back to `s` using the `parent` array. The bottleneck is the minimum residual capacity along the path.
  </Step>
  <Step title="Update the residual graph">
    For each edge `(u, v)` on the path, subtract the bottleneck from `rGraph[u][v]` and add it to `rGraph[v][u]`.
  </Step>
  <Step title="Accumulate flow">
    Add the bottleneck to `max_flow`. Return to step 2.
  </Step>
</Steps>

## BFS-based augmenting path search

The `breadthFirstSearch` function in `verifyMaxFlux.ts` implements step 2. It explores nodes layer by layer from the source, recording each node's predecessor in the `parent` array so the path can be reconstructed.

```typescript verifyMaxFlux.ts
function breadthFirstSearch(
  rGraph: number[][],
  s: number,
  t: number,
  parent: number[],
  V: number
) {
  const visited = new Array(V);
  for (let i = 0; i < V; ++i)
    visited[i] = false;

  const queue: number[] = [];
  queue.push(s);
  visited[s] = true;
  parent[s] = -1;

  while (queue.length != 0) {
    const u: number = queue.shift() || 0;

    for (let v = 0; v < V; v++) {
      if (visited[v] == false && rGraph[u][v] > 0) {
        if (v == t) {
          parent[v] = u;
          return true;
        }
        queue.push(v);
        parent[v] = u;
        visited[v] = true;
      }
    }
  }

  return false;
}
```

BFS guarantees that the first path found to `t` is the **shortest** (fewest edges). This is the key property of Edmonds-Karp: shortest augmenting paths prevent the O(E · max_flow) worst case that can occur with DFS-based path selection.

## Residual graph update

After the bottleneck `path_flow` is calculated, the residual graph is updated for every edge on the augmenting path:

```typescript verifyMaxFlux.ts
rGraph[u][v] = rGraph[u][v] - path_flow;
rGraph[v][u] = rGraph[v][u] + path_flow;
```

Subtracting from `rGraph[u][v]` reduces the remaining forward capacity. Adding to `rGraph[v][u]` creates (or increases) a backward edge that lets the algorithm cancel previously committed flow if a better global routing is found later.

## Step-by-step visualization with `stepStop`

The `fordFulkerson` function accepts an optional `stepStop` parameter. When provided, the algorithm pauses after the edge update at that specific step index and returns the intermediate state of the residual graph along with a `finished: false` flag.

```typescript verifyMaxFlux.ts
if (
  stepStop != undefined &&
  stepStop != null &&
  stepStop === currentStep
)
  return { maxFlow: max_flow, rGraph, finished: false };

currentStep++;
```

The outer `while` loop increments `currentStep` for each edge processed inside an augmenting path. This lets the UI replay the algorithm one edge update at a time, showing exactly how the residual graph evolves.

<Note>
When `stepStop` is `null` or `undefined`, the algorithm runs to completion and returns `finished: true` along with the final maximum flow.
</Note>

## Time complexity

The Edmonds-Karp variant runs in **O(VE²)** time, where `V` is the number of vertices and `E` is the number of edges.

| Factor | Explanation |
|---|---|
| O(VE) augmenting paths | Shortest-path augmentation guarantees at most O(VE) iterations before termination |
| O(E) per BFS | Each BFS traverses at most all edges once |
| O(VE²) total | Product of the two bounds above |

This bound is independent of edge capacities, which makes Edmonds-Karp strongly polynomial.

## Worked example

Consider the graph encoded as:

```json
[[0,10,10,0],[0,0,5,15],[0,0,0,10],[0,0,0,0]]
```

This is a 4-node graph: source `s` (0), nodes 1 and 2, and sink `t` (3).

```
s --10--> 1 --5-->
|               \--> t
s --10--> 2 --10--> t
          1 --15--> t
```

**Iteration 1** — BFS finds `s → 1 → t` (shortest path, 2 hops via node 1 then edge 1→3 with capacity 15). Bottleneck = min(10, 15) = **10**. Flow becomes 10. Residual: `rGraph[0][1]` = 0, `rGraph[1][3]` = 5, backward edges updated.

**Iteration 2** — `s → 1 → t` is exhausted (`rGraph[0][1]` = 0). BFS finds `s → 2 → t`. Bottleneck = min(10, 10) = **10**. Flow becomes 20. Residual: `rGraph[0][2]` = 0, `rGraph[2][3]` = 0.

**Iteration 3** — BFS finds `s → 2 → 1 → t` using the backward edge `2 → 1` (added in iteration 2 via `rGraph[1][2]` updates — wait, node 1→2 has capacity 5). Actually BFS finds no further path from `s` since `rGraph[0][1]` = 0 and `rGraph[0][2]` = 0. No more augmenting paths.

**Result**: `max_flow = 20`.

<Tip>
You can verify this yourself by pasting `[[0,10,10,0],[0,0,5,15],[0,0,0,10],[0,0,0,0]]` into the generator's input field and observing the step-by-step visualization.
</Tip>
